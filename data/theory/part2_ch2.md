# 2과목 2장: 분석 마스터플랜

## 1. 분석 마스터플랜 수립 개요

* **정의**: 분석 마스터플랜은 기업 및 조직이 당면한 전략적 목표를 달성하기 위해 다수의 분석 과제를 빠짐없이 도출하고, 이에 대한 적용 우선순위를 설정하여 단기 및 중·장기 이행 로드맵을 체계적으로 수립하는 전략적 기획 활동이다.
* **ISP와의 관계**: 일반적인 정보전략계획(ISP, Information Strategy Planning) 방법론을 활용하되, 데이터 분석 기획만의 특수성(데이터 가용성, 분석 기법의 불확실성 등)을 유기적으로 반영하여 마스터플랜을 수립해야 한다.
  * **ISP (Information Strategy Planning)**: 정보기술 또는 정보시스템을 전략적으로 활용하기 위하여 조직 내외부 환경을 분석해 기회나 문제점을 도출하고, 사용자의 요구사항을 분석하여 시스템 구축 우선순위를 결정하는 등 중장기 마스터플랜을 수립하는 절차이다.
* **목표**: 일회성 분석 프로젝트에 그치지 않고, 기업의 업무 프로세스에 분석을 내재화하여 지속 가능한 분석 관리 체계(분석 거버넌스) 구축을 지향한다.

---

## 2. 마스터플랜 수립 프레임워크

분석 마스터플랜을 수립하는 전체 일련의 구조적 형식을 의미하며, 우선순위를 정하는 것으로부터 시작된다.

| 단계 | 주요 수행 내용 | 핵심 키워드 및 산출물 |
|---|---|---|
| **1. 적용 우선순위 설정** | 업무 영역별로 도출된 분석 후보 과제들을 기업의 가치 기준에 따라 평가하고 실행 순서를 결정 | 전략적 중요도, 비즈니스 성과/ROI, 실행 용이성 |
| **2. 분석 구현 로드맵 수립** | 선정된 과제들의 선·후행 관계를 정렬하고, 단계별 추진 목표와 타임라인을 설계 | 업무 내재화 수준, 분석 데이터 수준, 기술 적용 수준 |

## 3. 우선순위 평가 기준 및 포트폴리오 사분면 분석

### (1) 빅데이터 특징을 고려한 분석 ROI 관점

일반적인 IT 프로젝트가 전략적 중요도와 실행 용이성을 기준으로 삼는다면, 빅데이터 분석 과제는 빅데이터의 핵심 특징인 4V를 투자 비용(Investment)과 비즈니스 효과(Return)라는 ROI(Return on Investment) 관점으로 변환하여 평가한다.

* **투자 비용 요소 (Investment = 3V)**
  * **크기 (Volume)**: 데이터의 규모 및 방대한 양을 저장·처리·관리하기 위한 하드웨어·인프라 투자 비용
  * **다양성 (Variety)**: 정형·반정형·비정형 등 다양한 형태를 가진 데이터를 입수하고 정제하는 데 필요한 투자 비용
  * **속도 (Velocity)**: 실시간 생성 데이터를 빠르게 가공·분석하기 위해 요구되는 기술적 인프라 및 가속화 비용
* **비즈니스 효과 요소 (Return = 1V)**
  * **가치 (Value)**: 데이터 분석을 통해 추구하거나 달성하고자 하는 기업의 비즈니스 목표 가치 및 실질적 효과

### (2) ROI 요소를 활용한 핵심 평가 축: 시급성과 난이도

* **시급성 (Urgency)** — '비즈니스 효과(Value)'와 직결
  * **판단 기준**: 전략적 중요도와 목표가치(KPI)
  * **설명**: 현재 시점에서 이 과제를 추진하는 것이 기업의 핵심 전략과 얼마나 정렬되어 있는지, 당면한 비즈니스 문제를 해결하기 위해 얼마나 시급한지를 평가
* **난이도 (Feasibility)** — '투자 비용 요소(3V)'와 직결
  * **판단 기준**: 데이터 획득·저장·가공 비용, 분석 적용 비용, 분석 수준(기술 복잡도)
  * **설명**: 해당 분석 과제를 기술적으로 구현하고 현업 시스템에 적용하는 데 소요되는 자원과 비용적 장벽을 평가

### (3) 포트폴리오 사분면 분석을 통한 과제 선정

우선순위 평가 결과를 바탕으로 시급성(현재/미래)과 난이도(Difficult/Easy)의 사분면 매트릭스에 과제를 배치하여 실행 순서를 결정한다.

```
       [난이도: Difficult]
               │
        Ⅰ 영역 │  Ⅱ 영역
               │
현재 ──────────┼────────── 미래 [시급성]
               │
       Ⅲ 영역  │  Ⅳ 영역
               │
         [난이도: Easy]
```

* **사분면별 특징 및 실행 방향**
  * **Ⅰ 영역 (현재-Difficult)**: 전략적 중요도가 높아 시급하게 추진할 필요가 있으나 난이도가 높아 바로 적용하기 어려움. 데이터 양, 분석 범위 등의 난이도를 조정하여 실행 순서를 앞당길 수 있음
  * **Ⅱ 영역 (미래-Difficult)**: 비즈니스 가치가 장기적 관점에 있고 구현 난이도도 매우 높아 중·장기적 관점에서 점진적으로 추진해야 함
  * **Ⅲ 영역 (현재-Easy)**: 시급성도 높고 난이도도 낮아 가장 우선적으로 실행(Quick-Win)하여 단기적인 성공 체험을 확보해야 함
  * **Ⅳ 영역 (미래-Easy)**: 과제 적용은 어렵지 않으나 당장 전략적 중요도가 높지 않아 중·장기적 관점에서 추진하거나 후순위로 배치함

* **의사결정 기준에 따른 추진 순서**
  1. **시급성 관점 우선순위**: Ⅲ 영역 → Ⅳ 영역 → Ⅱ 영역 순서로 진행 (Ⅰ 영역은 난이도가 높아 시급성 조율 대상이 됨)
  2. **난이도 관점 우선순위**: Ⅲ 영역 → Ⅰ 영역 → Ⅱ 영역 순서로 진행 (쉬운 것에서 어려운 것 순서로 전개)

> **핵심**: 시급성 관점은 Ⅲ→Ⅳ→Ⅱ, 난이도 관점은 Ⅲ→Ⅰ→Ⅱ 순서로 진행된다는 점을 혼동하지 않아야 한다. 두 관점 모두 Ⅲ 영역(Quick-Win)에서 시작하지만, 그다음 갈림길에서 시급성 관점은 Ⅳ 영역(미래·쉬움)을, 난이도 관점은 Ⅰ 영역(현재·어려움)을 우선한다.

## 4. 분석 구현 로드맵 및 이행 계획 수립

### (1) 로드맵 수립 단계의 3대 고려 요소

포트폴리오 사분면 분석으로 1차 우선순위가 결정되면, 최종 로드맵 확정을 위해 분석 과제의 적용 범위 및 방식을 세부적으로 설정해야 한다.

* **업무 내재화 적용 수준**: 분석 결과를 현업 업무 프로세스에 얼마나 깊숙이 동기화하고 자동화할 것인지 설정
* **분석 데이터 적용 수준**: 내부의 기본적인 정형 데이터로 한정할 것인지, 외부의 비정형·반정형 데이터 영역까지 점진적으로 확대할 것인지 설정
* **기술 적용 수준**: 기존 통계 패키지 활용 수준에서 고급 머신러닝·AI 알고리즘 및 빅데이터 분산 처리 기술로 고도화할 것인지 설정

> **핵심**: "~수준"이 붙은 세 요소(업무 내재화, 분석 데이터, 기술 적용 수준)는 **로드맵 수립 시 고려 요소**이고, 앞서 다룬 전략적 중요도·비즈니스 성과/ROI·실행 용이성은 **우선순위 설정 기준**이다. 두 그룹을 서로 바꿔 묻는 문제에 주의한다.

### (2) 세부 이행 계획 수립 시 고려사항: 폭포수형 vs 나선형(반복형)

* **고전적인 폭포수(Waterfall) 방식의 한계**: 철저히 순차적으로만 진행되는 폭포수 방식은 데이터 분석 프로젝트의 특성(불확실성, 피드백 순환)을 수용하기 어렵다.
* **나선형 모델(Spiral Model) — 반복적 정련(Iterative Refinement) 일정 계획**: 분석 프로젝트는 데이터 탐색과 모델링 단계에서 수많은 시행착오와 반복 수행(Feedback Loop)이 빈번히 발생하므로, 위험 분석(Risk Analysis)을 매 반복 주기(iteration)마다 수행하며 점진적으로 완성도를 높여가는 나선형 모델 방식으로 세부 이행 일정을 유연하게 수립해야 한다.

> **핵심**: 분석 마스터플랜의 세부 이행계획은 순차적인 폭포수 모델이 아니라, 반복(Iteration)마다 위험을 재점검하며 정련해 나가는 **나선형 모델**(Spiral Model)을 채택한다는 점이 자주 출제된다. "반복적 정련"과 "나선형 모델"은 같은 개념을 가리키는 표현이다.

## 5. 데이터 분석 수준 진단

지속 가능한 데이터 분석 역량을 갖추기 위해 기업은 무작정 솔루션을 도입하기 전, 현재 자신의 분석 기반과 기술 수준이 어디에 위치해 있는지 객관적인 진단을 수행해야 한다. 진단은 **분석 준비도**와 **분석 성숙도**의 두 가지 축으로 구성된다.

### (1) 분석 준비도 (Readiness)

기업이 데이터 분석을 도입하고 성공적으로 실행할 수 있는 역량과 내부 인프라 환경을 갖추었는지 진단하는 지표로, 총 6개 영역으로 구성된다.

| 진단 영역 | 주요 평가 및 체크 항목 |
|---|---|
| **1. 분석 업무 파악** | 발생한 사실 여부 파악, 예측 분석 업무, 시뮬레이션 분석 업무, 최적화 분석 업무, 분석 업무의 정기적 개선 활동 여부 |
| **2. 인력 및 조직** | 분석 전문가 직무 존재 여부, 분석 전문가 교육 훈련 프로그램 운영, 관리자들의 기본적 분석 능력 수준, 전사 분석 업무 총괄 조직 존재, 경영진의 분석 업무 이해 능력 |
| **3. 분석 기법** | 업무별 적합한 분석 기법 사용, 분석 업무 도입 방법론 보유 여부, 분석 기법 라이브러리 관리, 분석 기법 효과성 평가 체계, 분석 기법의 정기적 개선 |
| **4. 분석 데이터** | 분석 업무를 위한 데이터 충분성, 데이터의 신뢰성, 데이터의 적시성 확보 여부, 비구조적(비정형) 데이터 관리 체계, 외부 데이터 활용 체계, 기준 데이터(Master Data) 관리 |
| **5. 분석 문화** | 사실에 근거한 의사결정 체계, 관리자의 데이터 중시 성향, 회의 등에서의 데이터 적극 활용, 경영진의 직관보다 데이터 우선 배려, 데이터 공유 및 협업 문화 정착 여부 |
| **6. IT 인프라** | 운영 시스템 데이터 통합 수준, EAI/ETL 등 데이터 유통 체계, 분석 전용 서버 및 스토리지 인프라, 빅데이터 분석 환경, 통계 분석 환경, 비주얼(시각화) 분석 환경 구축 여부 |

> **핵심**: 분석 예산·투자 비용은 이 6대 영역에 포함되지 않는다. 시험에서 단골 오답 선지로 등장하니 주의한다.

### (2) 분석 성숙도 모델 (Maturity)

소프트웨어 공학의 CMMI(Capability Maturity Model Integration) 모델을 기반으로 구축되었으며, 조직의 데이터 분석 역량과 활용 결과가 얼마나 성숙하게 운영되고 있는지를 진단한다.

* **진단 대상 3개 부문**

| 진단 부문 | 평가 내용 |
|---|---|
| **비즈니스 부문** | 발생한 사실 분석·예측 분석·시뮬레이션·최적화 등 분석 결과가 실제 업무·의사결정에 얼마나 폭넓게 활용되고 있는지 |
| **조직·역량 부문** | 분석 전담 조직 존재 여부, 전문 인력 보유 수준, 부서 간 협업 및 분석 문화의 성숙도 |
| **IT 부문** | 데이터 관리 체계, 분석 전용 인프라·플랫폼, 시각화·분석 환경 등 기술적 기반의 수준 |

* **성숙도 수준 분류 (4단계)**
  1. **도입 단계**: 분석을 처음으로 시작하여 기본적인 데이터 분석 환경과 시스템을 구축하는 단계
  2. **활용 단계**: 구축된 인프라를 통해 도출된 분석 결과를 실제 업무 프로세스에 적용하고 실무에 활용하는 단계
  3. **확산 단계**: 특정 부서의 성과를 넘어 분석 결과를 전사 차원에서 체계적으로 관리하고 공유하는 단계
  4. **최적화 단계**: 데이터 분석이 일상화되어 경영 의사결정의 핵심으로 자리 잡고, 분석을 통해 지속적인 비즈니스 혁신 및 전사적 성과 향상에 직접적으로 기여하는 최고 수준의 단계

### (3) 분석 수준 진단 결과에 따른 기업 유형 분류

분석 준비도(6대 영역)와 분석 성숙도(4단계)를 두 축으로 결합하면 기업의 현 상태를 4가지 유형으로 매핑하여 맞춤형 개선 방안을 수립할 수 있다.

| 유형명 | 진단 결과 (준비도 & 성숙도) | 특징 및 맞춤형 개선 방향 |
|---|---|---|
| **준비형** | 낮은 준비도 & 낮은 성숙도 | 분석을 위한 데이터, 조직, 인력, 기법 등 모든 구성요소가 적용되어 있지 않아 가장 기초적인 사전 준비 및 기반 수립이 시급한 기업 |
| **정착형** | 낮은 준비도 & 높은 성숙도 | 인력이나 전사 인프라 같은 준비도는 낮으나, 특정 업무 부서 중심으로 분석 능력을 제한적으로 활용하고 있어 1차적으로 분석 문화의 사내 정착이 시급한 기업 |
| **도입형** | 높은 준비도 & 낮은 성숙도 | 현업의 분석 업무 활용이나 성숙도는 낮으나, 고급 인력 및 전사 조직, 분석 서버 등 준비도가 대단히 높아 데이터 분석을 바로 본격적으로 도입할 수 있는 기업 |
| **확산형** | 높은 준비도 & 높은 성숙도 | 데이터 분석을 위한 6가지 구성요소를 완벽히 갖추고 전사적으로 활용 중이며, 향후 지속적인 고도화와 전사적 확산 및 최적화가 가능한 선도적 기업 |

## 6. 분석 거버넌스 체계 수립

### (1) 분석 거버넌스(Analytics Governance)의 개념

* 기업 내에서 데이터 분석 활동이 일회성으로 끝나지 않고 조직의 문화로 정착되고 지속적으로 고도화될 수 있도록, 데이터와 분석 과정을 관리·유지·규제하는 프레임워크이자 표준화된 내부 관리 체계를 의미한다.
* **분석 거버넌스 체계의 5대 구성 요소**: 조직 및 인력(Organization & Human Resource), 과제 기획 및 운영 프로세스(Process), 분석 관련 시스템 및 인프라(System), 데이터(Data), 분석 관련 교육 및 마인드 육성 체계(변화관리)
* **분석 지원 인프라 (플랫폼)**
  * **의미**: 단순한 분석 응용프로그램뿐만 아니라 분석 서비스를 위한 응용프로그램이 실행될 수 있는 기초를 이루는 컴퓨터 시스템
  * **역할**: 하드웨어에 탑재되어 데이터 분석에 필요한 프로그래밍 환경과 실행 및 서비스 환경을 제공

> **핵심**: 분석 비용 및 과제 예산은 거버넌스 체계의 5대 구성 요소에 포함되지 않는다.

### (2) 데이터 거버넌스(Data Governance) 체계 수립

데이터 거버넌스는 전사 차원의 모든 데이터에 대하여 정책 및 지침, 표준화, 운영 조직을 구축하고 데이터의 가용성·통합성·보안성을 확보하는 체계로, 분석 거버넌스를 완벽히 구현하기 위한 핵심 토대가 된다.

* **데이터 거버넌스의 3대 요소**: 원칙(Principle), 조직(Organization), 프로세스(Process)
* **데이터 거버넌스의 3대 중요 관리 대상**
  1. **마스터 데이터 (Master Data)**: 비즈니스 트랜잭션의 기본이 되는 자료 집합으로, 자주 변하지 않고 시스템 전반에 기본 정보를 제공하는 핵심 데이터
  2. **메타데이터 (Meta Data)**: 데이터에 대한 데이터로, 데이터의 속성·출처·관계 등을 구조적으로 기술한 정보
  3. **데이터 사전 (Data Dictionary)**: 데이터베이스 내 저장된 정보의 요약이자 설명서로, 데이터 해석의 오류와 혼선을 제거하기 위해 구축

* **데이터 거버넌스 체계의 4가지 구성 요소**
  1. **데이터 표준화**: 데이터 표준 용어 설정, 명명 규칙(Naming Rule) 수립, 메타데이터 및 데이터 사전 구축 업무
  2. **데이터 관리 체계**: 표준 데이터와 메타데이터의 관리 원칙 수립, 데이터 품질 관리 프로세스 정의, 접근 권한 및 보안 정책 수립
  3. **데이터 저장소 관리 (Repository)**: 메타데이터 및 표준 데이터를 전사 차원에서 통합 관리하기 위한 저장소를 설계 및 운영하며, 데이터 구조 변경 시 사전 영향 평가를 수행함
  4. **표준화 활동**: 데이터 거버넌스 구축 후 준수 여부를 주기적으로 점검하고 모니터링을 실시하며, 안정적 정착을 위한 변화 관리 및 주기적 임직원 교육을 진행함

### (3) 데이터 분석 조직의 구조 유형

데이터 분석 업무를 수행하는 주체와 전담 인력의 배치 방식에 따라 조직 구조는 크게 3가지 형태로 나뉜다.

#### 1) 집중 구조 (Centralized Structure)

* **특징**: 전사 분석 업무를 별도의 독립적인 분석 전담 조직에서 총괄하여 담당함
* **장점**: 전사 전략적 중요도에 따라 분석 조직이 자체적으로 우선순위를 명확히 정해서 리소스를 집중 투입할 수 있음
* **단점**: 현업 부서의 고유 업무와 밀착되기 어려워, 현업의 요구사항과 분석 업무 간의 이중화 및 이원화 가능성이 매우 높음

#### 2) 기능 구조 (Functional Structure)

* **특징**: 별도의 분석 전담 조직이 존재하지 않으며, 일반적인 부서 구조 하에서 해당 현업 업무 부서가 직접 필요한 분석을 수행함
* **장점**: 현업 부서의 실무 요구사항이 즉각 반영되며 업무 밀착도가 높음
* **단점**: 전사적 관점의 핵심 분석이나 유기적 데이터 통합 분석이 불가능하며, 부서별 현황 통계 등 과거 실적에 국한된 분석에 그칠 가능성이 높음

#### 3) 분산 구조 (Decentralized Structure)

* **특징**: 전사 분석 전담 조직 소속의 분석 인력들을 각 현업 부서로 직접 배치하여 현업 부서 내에서 분석 업무를 수행하도록 함
* **장점**: 전사 차원의 우선순위 통제와 동시에 분석 결과를 현업에 신속하게 적용(Action)할 수 있으며, 부서 간 베스트 프랙티스 공유가 용이함
* **단점**: 현업 부서의 고유 업무와 분석 업무 간의 역할 분담을 명확히 하지 않으면, 분석 인력에게 현업 업무 과다로 인한 업무 이원화 및 정체성 혼선이 발생할 수 있음

### (4) 분석 과제 관리 프로세스 수립

* **과제 발굴 단계**: 개별 조직이나 개인이 도출한 분석 아이디어를 발굴하고 이를 과제화하여 분석 과제 풀(Pool)로 관리하면서 분석 프로젝트를 선정하는 작업을 수행한다.
* **과제 수행 단계**: 분석을 수행할 팀을 구성하고 분석 과제 실행 시 지속적인 모니터링과 과제 결과를 공유하고 개선하는 절차를 수행한다.

### (5) 분석 조직 인력 구성 시 고려사항

* 분석 조직을 설계할 때는 비즈니스 질문(Question)을 사전에 선제적으로 발굴할 수 있는 구조인지 점검해야 한다.
* 조직 내 인력 구성은 비즈니스 인력, IT 기술 인력, 분석 전문 인력, 변화관리 인력, 교육 담당 인력으로 나뉜다. 거버넌스의 안정적 정착을 위한 교육 및 조직문화 변화관리 담당 인력을 포함해야 한다.
