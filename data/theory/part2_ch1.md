# 2과목 1장: 데이터분석 기획의 이해

## 1. 분석 기획의 정의와 특징

### (1) 분석 기획의 정의

분석 기획은 실제 분석을 수행하기 전, 어떠한 목표(**What**)를 달성하기 위해(**Why**) 어떠한 데이터를 가지고 어떠한 방식(**How**)으로 수행할 것인가에 대한 일련의 계획을 수립하는 사전 준비 작업이다. 데이터 분석 프로젝트의 성패를 좌우하는 핵심 출발점이다.

### (2) 목표 시점별 기획 방안 (단기 vs 장기)

분석 기획을 바라보는 관점은 시점에 따라 이원화하여 구분해야 한다.

| 구분 | 과제 중심적 접근 방식 (단기) | 장기적 마스터플랜 방식 (장기) |
|---|---|---|
| **목표** | 당면한 분석 주제를 빠르게 해결 | 지속적이고 체계적인 분석 문화 내재화 |
| **분석 단위** | 과제 단위 (Task/Project) | 마스터플랜 단위 (Master Plan) |
| **핵심 가치** | **Speed & Test** (신속성 및 검증) | **Accuracy & Deploy** (정확성 및 전개) |
| **과제 유형** | Quick-Win (즉각적인 성과 도출) | Long-Term View (장기적 관점) |
| **접근 방식** | **Problem Solving** (문제 해결 중심) | **Problem Definition** (문제 정의 중심) |

> **핵심**: 두 방식은 상호 배타적이지 않으며, 시급한 과제는 과제 중심적 접근으로 먼저 해결하고 이를 마스터플랜에 반영하여 중장기 로드맵과 병행하는 것이 바람직하다.

### (3) 분석 주제 유형 (4가지 사분면)

분석 기획 시 분석의 대상(What)과 분석의 방법(How)이 인지(Known)되어 있는지, 미지(Unknown)인지에 따라 4가지 유형으로 나뉜다.

| | 방법(How)을 아는 경우 | 방법(How)을 모르는 경우 |
|---|---|---|
| **대상(What)을 아는 경우** | 최적화 (Optimization) | 솔루션 (Solution) |
| **대상(What)을 모르는 경우** | 통찰 (Insight) | 발견 (Discovery) |

1. **최적화 (Optimization)**: 분석 대상과 방법이 모두 명확한 상황이며, 효율성 극대화 및 비용 최소화를 목표로 한다.
2. **솔루션 (Solution)**: 분석 대상은 명확하게 알고 있으나, 구체적인 분석 방법을 모를 때 해결책을 탐색하는 방식이다.
3. **통찰 (Insight)**: 분석 방법은 이미 알고 있으나, 분석 대상(What)이 불명확할 때 기존 방법을 적용해 새로운 패턴을 찾아 통찰을 얻는 방식이다.
4. **발견 (Discovery)**: 분석 대상과 분석 방법 모두 불명확한 상황에서 새로운 데이터와 관계를 탐구하여 문제를 새롭게 발견하고 정의하는 방식이다.

> **핵심**: 대상과 방법을 모두 알면 최적화, 대상만 알고 방법을 모르면 솔루션, 방법만 알고 대상을 모르면 통찰, 둘 다 모르면 발견이다. "대상은 모르지만 기존 방법을 적용"하는 경우를 발견으로 착각하지 않도록 주의한다 — 이는 통찰(Insight)에 해당한다.

## 2. 분석 기획 시 고려사항

성공적인 분석 기획을 위해서는 데이터, 유즈케이스, 장애요소라는 3대 축을 검토해야 한다.

| 고려사항 | 세부 내용 |
|---|---|
| **① 가용한 데이터 확인**(Available Data) | 데이터의 유무뿐만 아니라 데이터의 유형(정형·반정형·비정형)에 따라 적용 가능한 분석 방법과 저장 인프라 솔루션이 달라지므로 이를 먼저 파악하는 것이 분석의 첫 단추이다. *참고*: 정형 데이터는 수직적 확장이 가능한 **RDB**, 반정형·비정형 데이터는 분산 저장에 유리한 **NoSQL**을 주로 사용한다. |
| **② 적절한 유스케이스 발굴**(Proper Use-Case) | 비즈니스 목표와 분석 목표를 일치시키기 위해 유사 업종·부서의 분석 사례(Use-Case)를 탐색하고 벤치마킹한다. 발굴된 유스케이스는 분석을 구체화하는 가이드라인 역할을 수행한다. |
| **③ 장애 요소에 대한 사전 계획**(Barriers) | 분석 프로세스 진행 중 발생할 수 있는 잠재적 장애요소(리스크)를 식별한다. 기술적 문제뿐만 아니라 조직 문화, 인력의 역량 부족, 예산 한계 등 비기술적 장애요소에 대한 사전 대응책을 수립해야 프로젝트의 좌초를 막을 수 있다. |

## 3. 데이터 분석 방법론의 이해

### (1) 데이터 분석 방법론의 계층 구조

분석 방법론은 체계적이고 반복적인 수행을 위해 **단계, 태스크, 스텝**의 3계층 프로세스 모델로 구성된다.

- **단계 (Phase)**: 최상위 계층으로, 큰 흐름을 정의한다. 각 단계는 기준선(Baseline)으로 설정되어 통제되며 변경 시 버전 관리를 거친다.
- **태스크 (Task)**: 단계를 구성하는 세부 활동 단위로, 품질 검토의 논리적·물리적 단위가 된다.
- **스텝 (Step)**: 태스크를 더 세분화한 최하위 실행 단위로, WBS의 워크 패키지에 해당하며 입력(Input)·처리(Process)·출력(Output) 구조를 갖는다.

### (2) KDD (Knowledge Discovery in Databases) 방법론

1996년 등장한, 데이터베이스에서 유용한 지식을 발견하기 위한 통계 중심의 체계적인 분석 절차이다. 총 **5단계**로 구성된다.

```
비즈니스 이해/목표 설정 → 데이터 선택 → 데이터 전처리 → 데이터 변환 → 데이터 마이닝 → 해석 및 평가
```

1. **비즈니스 이해와 목표 설정**: 도메인 지식을 바탕으로 분석 목적을 수립한다.
2. **데이터 선택 (Selection)**: 대상 데이터베이스에서 분석 목적에 맞는 데이터셋을 선정한다.
3. **데이터 전처리 (Preprocessing)**: 데이터의 신뢰성을 높이기 위해 잡음(Noise), 결측치(Missing Value), 이상치(Outlier)를 정제하는 단계이다.
4. **데이터 변환 (Transformation)**: 분석 목적에 맞게 데이터의 형태를 바꾼다. 변수 선택(Feature Selection), 차원 축소(Dimensionality Reduction), 데이터 스케일링(Scaling) 등을 통해 과대적합을 방지하고 분석 효율을 높인다.
5. **데이터 마이닝 (Data Mining)**: 변환된 데이터셋에 수리 알고리즘을 적용하여 실제로 패턴을 발견하고 분류·예측을 수행하는 핵심 단계이다.
6. **해석 및 평가 (Interpretation/Evaluation)**: 도출된 마이닝 결과를 해석하고 비즈니스 목적에 부합하는지 최종 평가한다.

> **핵심**: 결측치·이상치·노이즈 정제는 **전처리(Preprocessing)**, 변수 선택·차원 축소·스케일링은 **변환(Transformation)** 단계에서 수행한다는 점을 혼동하지 않아야 한다.

### (3) CRISP-DM 방법론

유럽 ESPRIT 프로젝트에서 시작되어 산업계 표준으로 자리 잡은 방법론으로, 총 **6단계**로 구성되며 각 단계 간 피드백과 반복·순환이 자유롭다.

1. **업무 이해 (Business Understanding)**: 비즈니스 관점에서 목표와 요구사항을 이해하고, 이를 데이터 분석 문제로 정의하는 첫 단계이다. (세부 태스크 예: 업무 목적 파악, 상황 파악, 데이터 마이닝 목표 설정)
2. **데이터 이해 (Data Understanding)**: 초기 데이터를 수집하고 속성을 파악하며 품질 문제를 식별하고 숨겨진 인사이트를 탐색한다.
3. **데이터 준비 (Data Preparation)**: 수집된 원본 데이터로부터 마이닝 알고리즘에 투입할 분석용 최종 데이터셋을 편성한다. (정제, 변환, 결합 등 포함)
4. **모델링 (Modeling)**: 다양한 데이터 마이닝 알고리즘을 선택해 파라미터를 최적화하고 모델을 생성한다. 모델링 기법 선택, 모델 테스트 계획 설계, 모델 작성이 이루어진다. (※ 모델의 업무 적용성 평가는 다음 단계인 '평가'에 속하므로 오답 함정에 주의)
5. **평가 (Evaluation)**: 생성된 모델이 비즈니스 목적과 목표를 충족하는지 검증한다. 분석 결과 평가, 모델링 과정 평가, 업무 적용성 평가가 함께 수행된다.
6. **전개 (Deployment)**: 완성된 모델을 운영 환경에 적용하고, 모니터링·유지보수 계획을 수립하며 프로젝트 종료 보고서를 작성한다.

### (4) KDD와 CRISP-DM 비교 매핑

두 방법론의 단계를 섞어놓고 선후 관계나 유사 단계를 묻는 문제가 자주 출제된다.

| KDD 단계 | CRISP-DM 단계 | 거시적 활동 분류 |
|---|---|---|
| 비즈니스 이해 → 데이터 선택 → 데이터 전처리 | 업무 이해 → 데이터 이해 | **초기 단계**: 비즈니스 목적 파악 및 데이터셋 선택·정제 |
| 데이터 변환 → 데이터 마이닝 | 데이터 준비 → 모델링 | **중간 단계**: 데이터 편성, 변환 및 마이닝 알고리즘 적용 |
| 해석 및 평가 | 평가 → 전개 | **최종 단계**: 지식 발견, 모델 평가 및 실제 업무 적용 |

### (5) SEMMA 방법론

SAS 통계 연구소가 예측 모델링에 특화하여 개발한 기술(통계) 중심 분석 방법론으로, 아래 5단계로 진행되는 순환적 절차이다.

```
Sample → Explore → Modify → Model → Assess
```

1. **Sample (표본추출)**: 모집단에서 분석에 적합한 크기의 표본을 추출한다. 표본이 모집단의 특성을 충분히 대표하면서도 분석 처리가 가능한 크기여야 한다.
2. **Explore (탐색)**: 추출된 표본 데이터에 대해 탐색적 데이터 분석(EDA)을 수행하여 변수 간 관계, 이상값, 패턴을 파악한다.
3. **Modify (수정·가공)**: 탐색 결과를 바탕으로 파생변수를 생성하거나 변수를 그룹화·변환하여 모델링에 적합한 형태로 데이터를 가공한다.
4. **Model (모델링)**: 목적에 맞는 다양한 데이터 마이닝 기법(회귀분석, 의사결정나무, 신경망 등)을 적용하여 모형을 구축한다.
5. **Assess (평가)**: 구축된 모델들의 예측력·안정성을 비교·평가하여 최종 모델을 선정한다.

> **핵심**: KDD·CRISP-DM은 데이터 마이닝 전반의 범용 프로세스인 반면, SEMMA는 통계적 예측 모델링에 특화된 절차라는 점에서 성격이 다르다. 세 방법론의 단계 수(KDD 5단계, CRISP-DM 6단계, SEMMA 5단계)를 서로 바꿔치기하는 오답에 주의한다.

## 4. 분석 과제 발굴 방법

분석 과제 발굴은 명확한 비즈니스 문제로부터 출발하는 **하향식 접근법**과 데이터 자체의 탐색을 통해 가치를 발견하는 **상향식 접근법**으로 나뉘며, 실무에서는 이 둘을 혼용한 **디자인 씽킹(Design Thinking)** 접근법을 함께 사용한다.

### (1) 하향식 접근법 (Top-Down Approach)

문제가 확실하게 주어졌을 때 사용하는 전통적인 방식으로, 각 과정이 체계적이고 단계적으로 진행된다.

```
문제 탐색 → 문제 정의 → 해결방안 탐색 → 타당성 검토 → 과제 선정
```

1. **문제 탐색 (Problem Discovery)**: 현재 비즈니스 상태를 분석하여 문제를 도출하는 단계로, 문제를 해결했을 때의 가치에 집중한다.
   - **비즈니스 모델 기반 탐색**: 비즈니스 모델의 5개 영역(업무, 제품, 고객, 규제와 감사, 지원 인프라)을 기준으로 과제를 발굴한다.
   - **외부 참조 모델 기반 탐색**: 유사·동종 업종의 분석 테마 후보 그룹을 참조하여 벤치마킹 형태로 문제를 탐색한다.
   - **분석 유즈 케이스(Analytics Use Case) 작성**: 도출된 분석 기회에 대한 상세 설명과 기대 효과를 문서화한다.
2. **문제 정의 (Problem Definition)**: 식별된 비즈니스 문제를 데이터 분석 문제로 변환하여, 필요한 데이터와 분석 기법을 재정의하는 핵심 단계이다.
3. **해결방안 탐색 (Solution Search)**: 정의된 분석 문제를 해결하기 위해 필요한 데이터, 분석 도구, 예산, 기술 인프라(RDB/NoSQL 등)를 검토하고 수행 옵션을 도출한다.
4. **타당성 검토 (Feasibility Study)**: 도출된 대안들의 실현 가능성을 평가한다.
   - **경제적 타당성**: 비용 대비 투자자본수익률(ROI), 비즈니스 가치 평가
   - **데이터 및 기술적 타당성**: 분석 역량, 가용 데이터 존재 여부, IT 인프라 여건 평가

### (2) 상향식 접근법 (Bottom-Up Approach)

문제 정의 자체가 불명확하거나 모르는 상태에서, 보유한 데이터 자체를 탐색하며 점진적으로 문제를 재정의하고 해결방안을 발견해 나가는 방식이다.

- **특징**: 반복적인 시행착오(Trial and Error)를 거치며 새로운 인사이트를 도출하는 데 매우 유용하다.
- **비지도 학습 기법 연계**: 사전에 기준이 되는 종속변수(정답)가 없는 상태에서 데이터의 군집·연관성을 파악하는 비지도 학습(Unsupervised Learning) 기법이 주로 활용된다.

### (3) 혼용 방법론: 디자인 씽킹 (Design Thinking)

하향식의 체계성(수렴, Convergence)과 상향식의 창의성(발산, Divergence)을 결합한 문제 해결 방법론이다. 문제를 탐색·확장하는 발산 단계와 문제를 정의·구체화하는 수렴 단계를 반복적으로 순환하며 혁신적인 분석 과제를 발굴한다.

## 5. 분석 프로젝트 관리 방안

분석 프로젝트는 결과의 불확실성이 높고 지속적인 피드백이 발생하므로, 일반 IT 프로젝트 관리 요소(일정, 범위, 자원 등) 외에 데이터 분석 특수성을 고려한 아래 5가지 관리 속성을 추가로 조율해야 한다.

| 주요 관리 속성 | 상세 설명 |
|---|---|
| **① Data Size (데이터 크기)** | 데이터의 양이 방대해질수록 연산 자원 확보와 분산 처리 시스템(하둡, NoSQL 등) 관리가 필수적이다. |
| **② Data Complexity (데이터 복잡성)** | 정형 데이터 외 텍스트·이미지·오디오 등 반정형·비정형 데이터가 섞여 있을 경우, 데이터 정제와 적합한 알고리즘 선정이 더 정교해져야 하므로 복잡성이 상승한다. |
| **③ Speed (분석 속도)** | 분석 결과를 도출하고 시스템에 반영해야 하는 시간적 요구사항이다. 실시간 스트리밍 처리가 필요한지, 주기적 배치 작업으로 가능한지에 따라 관리 방식이 다르다. |
| **④ Analytic Complexity (분석 복잡성)** | 모델의 정확도(Accuracy)를 높이기 위해 복잡한 알고리즘을 사용하면 모델의 해석력(Explainability)이 떨어지고 과적합 리스크가 발생한다. 정확도와 복잡성 간의 트레이드오프(Trade-off)를 조율해야 한다. |
| **⑤ Accuracy & Precision (정확도와 정밀도)** | **정확도**(Accuracy)는 실제 정답과 모델 예측값이 얼마나 일치하는가를, **정밀도**(Precision)는 모델이 예측한 값들이 얼마나 일정하게 모여있는가를 나타낸다. 분석 기획 단계에서 이 두 품질 기준을 명확히 설정해야 프로젝트 리스크를 방지할 수 있다. |

> **핵심**: 정확도와 정밀도는 항상 비례하지 않으며 트레이드오프 관계가 발생할 수 있다. 정확도는 참값과의 일치도, 정밀도는 결과의 일관성(재현성)을 의미한다는 차이를 구분해야 한다.
